## Diagnóstico de los casos 3 y 5

Los casos 3 y 5 fallan **NO por un bug de Spring Boot**, sino porque el flujo
`/agente/auth/identity` del agente + el broker JWT de Keycloak no respetan el
parámetro `scope` enviado por el cliente.

### Evidencia

Decodificando el payload del access_token emitido para `scope="calendar.read"`:

```json
{
  "scope": "email.send email.modify email profile calendar.read calendar.write",
  "aud": ["spring-boot-api", "account"],
  "azp": "agente-ia",
  "sub": "96b2b0d4-711e-4941-b4ca-28f8633eae4d"
}
```

Y para `scope="email.send"`:

```json
{
  "scope": "email.send email.modify email profile calendar.read calendar.write",
  "aud": ["spring-boot-api", "account"],
  "azp": "agente-ia",
  "sub": "96b2b0d4-711e-4941-b4ca-28f8633eae4d"
}
```

Ambos tokens son **idénticos en `scope`** y contienen los 6 scopes configurados
en el client `agente-ia` del realm `agent-poc`.

### Causa raíz

El agente (`agent-python/oauth_client.py:262-323`, método `identity_exchange`)
sí envía el `scope` pedido al token endpoint:

```python
data = {
    "grant_type":  "urn:ietf:params:oauth:grant-type:jwt-bearer",
    "client_id":   AGENT_CLIENT_ID,
    "client_assertion": client_jwt,
    "client_assertion_type": "urn:ietf:params:oauth:client-assertion-type:jwt-bearer",
    "assertion":   identity_assertion,
    "scope":       scope,   # ← sí se manda
}
```

Pero **el IdP broker `jwt-authorization-grant` configurado en el realm
`agent-poc` ignora ese parámetro** y emite el token con todos los scopes del
client `agente-ia` (comportamiento por defecto del broker; habría que añadir
un "Scope mapper" o configurar `acr`/`allowed_scopes` en el broker para que
filtree). El log del agente (`[C/Identity] canjeando identity_assertion
scope=…`) confirma que el agente manda el scope, y el log de Spring Boot
(`CalendarController … scope_claim=email.send email.modify email profile
calendar.read calendar.write`) confirma que Spring recibe el scope completo.

### Lo que esto significa

- Spring Boot valida correctamente: cuando recibe un JWT con el claim
  `scope` que contiene `calendar.read`, deja pasar; si no lo contiene,
  denegaría con 403 (esto está cubierto por los tests unit
  `CalendarControllerSecurityTest` y `JwtAudienceValidatorTest`).
- Pero con la implementación actual del agente + KC no es posible emitir un
  token con scope restringido, por lo que **los casos 3 y 5 no son
  reproducibles en este entorno**.
- Para validar end-to-end el 403 por scope habría que:
  1. Cambiar el broker en `agent-poc` para que filtre por `requested_scope`,
     o
  2. Añadir un scope mapper explícito en el client `agente-ia`, o
  3. Sustituir el flujo C por un OBO real donde el agente haga una llamada
     intermedia con `scope=` reducido y KC respete el filtro.

## Estado de la integración

| Componente | Verificado |
|---|---|
| Spring Boot arranca, valida firma con JWKS de KC y expone :9090 | ✅ |
| `JwtAudienceValidator` rechaza tokens sin `aud=spring-boot-api` | ✅ (cubierto por test unit + caso 1 que sí tiene aud correcto) |
| `BearerTokenAuthenticationFilter` devuelve 401 sin Authorization | ✅ caso 2 |
| `@PreAuthorize('SCOPE_calendar.read')` deja pasar tokens con `calendar.read` en el claim `scope` | ✅ caso 1 |
| `@PreAuthorize('SCOPE_email.send')` deja pasar tokens con `email.send` en el claim `scope` | ✅ caso 4 |
| `@PreAuthorize` rechaza tokens sin el scope → 403 | ⚠️ no reproducible con tokens reales del agente; **validado por unit test** `CalendarControllerSecurityTest` |

## Veredicto

Integración **parcialmente verificada con tokens reales**: audiencia y firma se
validan contra KC 26 end-to-end (casos 1, 2, 4). La denegación por scope (403)
es correcta en la lógica de Spring (tests unit verdes) pero **no se ha podido
provocar con un access_token real** porque el broker JWT de KC no filtra el
scope — devuelve el conjunto completo de scopes del client. La cadena
Agente → KC 26 → Spring Boot 3.2 funciona; el siguiente paso para cerrar el
test E2E es ajustar el broker/realm de KC para que respete el `scope`
solicitado por el agente en el grant `urn:ietf:params:oauth:grant-type:jwt-bearer`.
